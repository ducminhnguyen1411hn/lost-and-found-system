# Feature: Claim (return request) & two-way verified handover  (FR-CLAIM-01..05)

- **Dimension:** D-STATE-MACHINE (multi-actor lifecycle with transactional side-effects) + a first cut of the shared **Notification infra** (DB rows + navbar bell; SignalR deferred to FR-NOTI).
- **By:** Dev A   **Status:** Done (matching/SignalR/dispute deferred — see below). Reviewed (0 Critical; 3 Important fixed).
- **Touches:** `Claim`, `ClaimImage` (new, DB-First), `FoundItem` (handover flags), `Notification` · `ClaimService`, `NotificationService` (impl of `INotificationService` + new `INotificationQueries`) · `ClaimsController`, `NotificationsController` · ViewComponents `ItemClaimPanel`, `NotificationBell` · Views `Claims/Create|My`, `Notifications/Index`, partials `_ClaimForm`, `_NotificationBell` + `ItemClaimPanel/Default` · edits to `FoundItems/Details`, `_Layout`, `Program.cs` DI.

## A. Lessons & application
- **Business goal:** core value #2 of the app — a loser proves an `Open` found item is theirs (verification text + ≤5 evidence photos), the **holder** accepts exactly one claim (others auto-reject), and the item only becomes `Returned` when **both** sides confirm the physical handover. Everything audited + both parties notified.
- **Decisions worth remembering:**
  - **Claim UI decoupled via ViewComponents, not by touching `FoundItemService`.** `ItemClaimPanel` (invoked from `FoundItems/Details` with just `itemId`) calls `IClaimService.GetItemClaimPanelAsync` and computes `CanClaim` / holder-only claim list / handover buttons per viewer. `FoundItemDetailViewModel` was left untouched. Same trick for `NotificationBell` in `_Layout`. Keeps the found feature's files stable and the claim feature self-contained.
  - **Notifications = DB rows now, SignalR later.** `NotificationService` implements the shared `INotificationService` (`PushAsync`/`PushToStaffAsync`) by writing a `Notification` row (no self-transaction — it `SaveChanges` inside the caller's tx, exactly like `AuditService`). A second interface `INotificationQueries` (unread count / recent / mark-read) drives the bell. `// TODO (FR-NOTI)` seams mark where the SignalR push goes. This unblocked FR-CLAIM's DoD ("notify the other party") without owning Dev B's realtime layer.
  - **Holder derived from `HoldingType`, never role.** `IsHolderOrAdmin`: SelfHeld → `ReporterUserId`, Custodial → `CustodianStaffId`, plus Admin. (Today only SelfHeld items reach `Open` because FR-HOLD isn't built, but the derivation is already general.)
  - **Blind listing by construction:** `GetItemClaimPanelAsync` only *builds* `PendingClaims`/`AcceptedClaim` (which carry `VerificationDetails` + evidence URLs) when `isHolder`. Anonymous users, non-holder members, other claimants, and non-custodian staff literally never receive the data — not hidden by CSS.
- **Traps hit (most valuable — all caught by the final review, all fixed):**
  1. **Public timeline leaked the claimant's name.** The public item timeline (`FoundItemDetailViewModel.PublicEvents`) renders `· bởi @ActorName` for every `IsPublic=1` `AuditLog` row. `FinalizeHandoverAsync` was logging `HandoverConfirmed`/`Returned` with `actorUid` = the confirming user → when the *claimant* confirmed receipt, the anonymous-visible page publicly named who reclaimed the item. **Fix:** log those public milestone rows with an **empty actor** (`LogAsync("", ...)`); the Detail text already states the role ("Người nhận xác nhận…"), and accept/reject accountability still lives on `Claim.HandledByUserId`. Rule reaffirmed: *public audit rows carry no names, not just no verification text.*
  2. **Concurrent both-confirm could strand the item at `ClaimAccepted` forever.** `bothConfirmed` was computed from the `FoundItem` loaded *before* the transaction; each side only flipped its own flag, so if both confirmed near-simultaneously the DB ended with both flags `true` but `Status` never set to `Returned` (and both confirm buttons then vanish). **Fix:** inside the tx, `SaveChanges` this side's flag, then `await _db.Entry(item).ReloadAsync()` to read the authoritative flags (the `FoundItem` row lock serializes the two writers) before deciding `Returned`. Whoever commits second now completes the return.
  3. **`ImageUploadException` → 500 on claim submit.** Evidence upload throws `ImageUploadException` (extends `Exception`, **not** `InvalidOperationException`) for a bad type / >5 MB / Cloudinary error, but `ClaimsController.Create` only caught `InvalidOperationException`. **Fix:** `catch (Exception ex) when (ex is InvalidOperationException or ImageUploadException)` → friendly ModelState error (upload runs before the tx, so nothing is half-written). Mirror the `FoundItemsController` pattern when copying an upload flow.
  4. **`Claim` name clash:** `using System.Security.Claims;` + the `Claim` entity → `new Claim { }` is CS0104 ambiguous. Fully-qualify the entity at the instantiation site (`new LostAndFound.Models.Entities.Claim { }`).
- **Reusable where:** FR-CAM (staff request/respond state machine + `PushToStaffAsync`), FR-THANK (post-`Returned` action + notify), FR-MATCH (calls `INotificationService` on a match). The ViewComponent-panel pattern fits any "extra actions on someone else's item detail."

## B. Reusable recipe
- **Steps (in order):** 1) DB-First: add child table (`ClaimImage`) additively to `schema.sql`, migrate+drop the old single column, apply, `/db-rescaffold --table ClaimImage`, copy config into `ApplicationDbContext`, delete `ScaffoldDbContext`. 2) ViewModels. 3) `NotificationService` (+`INotificationQueries`) + DI. 4) Notifications UI (controller + bell ViewComponent + layout + Index). 5) `ClaimService` (whole state machine) + DI. 6) `ClaimsController`. 7) Claim UI (form + My + `ItemClaimPanel` + Details invoke). 8) build + run + drive + review.
- **Files created/changed:** `db/schema.sql` (+`ClaimImage`, drop `Claim.EvidenceImagePath`) · entities `ClaimImage`, `Claim` (regenerated) · `ApplicationDbContext` (DbSet+config) · VMs under `Models/ViewModels/{Claims,Notifications}` · `ClaimService`/`IClaimService`, `NotificationService`/`INotificationQueries` · `ClaimsController`, `NotificationsController` · 2 ViewComponents + views/partials · `Program.cs` DI (3 registrations).
- **Validation:** code = `[Required][StringLength]` on `VerificationDetails`, required `RejectReason`, ≤5 evidence + type/size in the upload service, **state guards in the service** (Open→claimable, Pending→accept/reject, ClaimAccepted→confirm/cancel, `Returned` only when both flags true, one active claim/user, no self-claim); DB = FK `Claim→FoundItem`, FK `ClaimImage→Claim` (cascade), `CK_Claim_Status (0..2)`.
- **Authorization:** `[Authorize(Roles="Member,Staff,Admin")]` on `ClaimsController`, `[Authorize]` on `NotificationsController`, `[ValidateAntiForgeryToken]` on every POST. Ownership enforced server-side in the service (holder-or-admin for accept/reject/handover/cancel; accepted-claimant for confirm-received) — hidden buttons are not the gate.
- **AuditLog:** `ClaimSubmitted`/`ClaimRejected` → `Claim`, **IsPublic=false** (identify people / carry the reason). `ClaimAccepted`/`AcceptanceCancelled` → `FoundItem`, **IsPublic=true** (actor = holder, whose name is already public). `HandoverConfirmed`/`Returned` → `FoundItem`, **IsPublic=true** but **actor empty** (never name the claimant). Public rows flow into the existing item timeline — no separate `_Timeline` built.
- **Notification:** submit→holder; accept→that claimant; auto-reject→each other claimant; reject→claimant(+reason); each handover confirm→the counterparty; `Returned`→both. All via `PushAsync` (DB row; bell shows it).
- **Minimum tests (verified):** build 0 errors; app boots (DI graph resolves); anon `Details` shows no claim button / no verification / no bell; `/Claims/My`,`/Notifications`,`/Claims/Create` → 302 login. **Still needs interactive 2-account drive** (submit→accept→2-sided handover→Returned; auto-reject of the loser; blind-listing for claimant vs holder; 6th evidence photo rejected).
- **Easy to get wrong when copying:** (a) public audit rows must not name the actor when it could be the claimant (trap #1); (b) decide multi-actor "both done?" from a DB re-read inside the tx, not a pre-tx in-memory copy (trap #2); (c) catch `ImageUploadException` on any upload POST (trap #3).
